## 1. LLM 컨텍스트 주입 아키텍처

### 1.1 현재 문제점

| 통합 지점 | 파일 | 현재 방식 | 문제 |
|----------|------|----------|------|
| 오늘의 운세 | `interpret-saju.ts` → `buildSajuContext()` | 사주 4기둥+오행+신살만 → 마크다운 | 십신/대운/형충파해 누락, 점성술·자미 없음 |
| 주간 운세 | 동일 | 동일 | 동일 |
| AI 채팅 | `chat/route.ts` → `buildContextBlock()` | birthInfo+sajuData+astroData → raw JSON 덤프 | 질문별 분기 없음, 구조화 안 됨 |
| 전문가 토론 | `run-debate.ts` | 각 에이전트 전용 빌더 | 자미두수 미참여, 크로스 연결 없음 |

**핵심 문제 6가지:**
1. 컨텍스트 빈곤 (십신·신강/신약·대운·형충파해 누락)
2. 점성술 미주입 (일일/주간 운세에 점성술 0%)
3. 자미두수 전무 (LLM에 한 번도 전달 안 됨)
4. 고전 DB 미활용 (토정비결·인생풀이 등 미주입)
5. 질문별 분기 없음 (모든 질문에 동일 컨텍스트)
6. 크로스시스템 연결 없음

### 1.2 제안 아키텍처 (To-Be)

```
User Query → IntentRouter → BundleAssembler → [Core + Primary + Enrichment] → LLM → 답변
                  ↓                  ↓
             질문 의도 분류      질문별 최적 데이터 선택 + 포맷팅
```

#### A. Intent Router (질문 라우터)

```
역할: 사용자 질문을 7개 의도 카테고리로 분류
방식: 키워드 매칭 (< 1ms) → 시맨틱 폴백 (선택)
위치: lib/fortune-agent/intent-router.ts

의도 분류 규칙:
- "성격|재능|약점|타입|MBTI" → IDENTITY
- "언제|시기|시점|때|시간" → TIMING
- "돈|재물|투자|사업|수입" → WEALTH
- "궁합|연애|결혼|관계|인연" → RELATIONSHIPS
- "건강|몸|운동|스트레스" → HEALTH
- "오늘|내일|이번주|지금" → DAILY
- "인생|전체|10년|미래|진로" → LIFE_DESIGN
- 디폴트(분류 불가) → DAILY
```

#### B. Bundle Assembler (데이터 번들 조립기)

```
역할: Intent에 따라 Core + Primary + Enrichment 데이터를 선택·포맷팅
위치: lib/fortune-agent/bundle-assembler.ts

입력: sajuResult + astrologyResult + ziweiBoard? + classicDB? + intent
출력: formatted markdown string (최적 포맷)

3계층:
  Core (~300 tokens, 항상): 일간, 십신, 대운, ASC, 명궁
  Primary (~500-800 tokens, intent별): 해당 분야 핵심 데이터
  Enrichment (~200-400 tokens, 선택): 크로스시스템 + 고전 DB
```

#### C. System Prompt Manager

```
역할: Intent별 최적 시스템 프롬프트 + DB 오버라이드
위치: lib/fortune-agent/system-prompts.ts

Intent별 해석 프레임워크:
  WEALTH: "재성의 위치·강도·흐름 중심 해석. 식상생재 여부, 대운에서 재성 출현 시기 집중."
  HEALTH: "오행 불균형에서 출발. 부족 오행의 관련 장기 언급. 의학적 진단 절대 금지."
  TIMING: "대운·세운·월운 시간축 위에 놓고 설명. 동서양 같은 방향이면 강조, 다르면 양면 설명."
```

### 1.3 토큰 예산

```
시스템 프롬프트:    ~500 tokens
Core Layer:        ~300 tokens  ← Prompt Caching 대상
Primary Layer:     ~500-800 tokens
Enrichment:        ~200-400 tokens
대화 히스토리:      ~1,000-2,000 tokens
사용자 질문:        ~100-200 tokens
─────────────────────────────
총 호출당:          ~2,600-4,200 tokens

→ Anthropic Prompt Caching: Core+System 캐시 (90% 비용 절감)
→ gpt-4o-mini 기준 충분히 수용 가능
```

### 1.4 캐싱 전략

```
레벨 1 — 클라이언트 캐시 (localStorage):
  saju_analysis_cache: birthInfo fingerprint별 무한 유효 ← 이미 있음
  astro_analysis_cache: 당일 TTL ← 이미 있음
  + ziwei_board_cache: birthInfo 기반 무한 유효
  + theme_interpretations_cache: birthInfo 기반 무한 유효
  + classic_db_cache: birthInfo 기반 무한 유효

레벨 2 — LLM Prompt Caching (Anthropic cache_control):
  System Prompt + Core Layer → cache_control: { type: "ephemeral" }
  동일 세션 내 반복 호출 시 90% input 비용 절감

레벨 3 — 서버 캐시 (향후):
  DivinationProfile 전체를 Redis에 캐시
  고정 데이터 = 24h TTL, 일일 데이터 = 자정까지
```

### 1.5 Pre-compute vs On-demand

| 데이터 | 전략 | 근거 |
|-------|------|-----|
| 사주 전체 (1-1~1-11) | Pre-compute | 고정 + 결정적 계산 |
| 점성술 natal (2-1~2-11) | Pre-compute | 고정 |
| 자미두수 명반 (5-1~5-3) | Pre-compute | 고정 |
| 테마별 해설 (3-1) × 10개 | Pre-compute | 가볍고 캐시 가능 |
| 고전 DB (3-3) × 10종 | Pre-compute | DB 조회, 캐시 |
| 점성술 today/transit (4-4, 4-7) | Daily refresh | 매일 변경 |
| 자미두수 일운/시운 (5-7, 5-8) | On-demand | 시간대별 변경 |
| Profection (4-9) | Yearly refresh | 연간 변경 |
| Dasha (4-10) | Pre-compute | 기간 고정 |
| Essential/Accidental Dignity | Daily refresh | 행성 위치 기반 |

## 4. 핵심 차별화 아이디어

### 4.1 삼합 리딩 (Triple-System Reading) ⭐⭐⭐

**컨셉**: 3시스템이 **같은 답** → 확신, **다른 답** → 다면적 해석

```
사용자: "올해 이직해도 될까요?"

[사주]    대운 편관 → "변화의 시기" (긍정)
```

## 12. Mastra 백엔드 아키텍처

### 12.1 권장 아키텍처

```
Client → Intent Classifier → Context Pack Router
              ↓                      ↓
         Mastra Workflow      Deterministic Tools
         Orchestrator         (Saju/Astro/Ziwei APIs)
              ↓
         Mastra Agents        Conflict-Synergy Engine
         (saju-master/            ↓
          astrologer/chat)   Response Composer
              ↓
         Memory Store         API Response + Evidence
         (resource/thread)
```

### 12.2 Intent → Context Pack 매핑

| Intent | 기본 Pack | 선택 Pack |
|--------|----------|----------|
| TODAY_GUIDE | P0_BASE, P1_DAILY | P6_CONFLICT |
| TIMING_WHEN | P0_BASE, P3_TIMING | P2_WEEKLY, P6 |
| DECISION_AB | P0_BASE, P1_DAILY | P3_TIMING, P7_ACTION |
| LOVE_RELATION | P0_BASE, P4_DOMAIN(love) | P5_RELATIONSHIP |
| CAREER_MONEY | P0_BASE, P4_DOMAIN(career/wealth) | P3_TIMING |
| WHY_EXPLAIN | P8_EVIDENCE | 모든 Pack 참조 |

### 12.3 Workflow 분해

| Workflow | 목적 | Steps |
|----------|------|-------|
| `reading-orchestrator-v1` | 일반 질문 | classify → loadPacks → detectConflict → generate → evidence |
| `decision-advisor-v1` | A/B 결정 | normalize → loadTiming → scoreAB → generate |
| `timeline-builder-v1` | 인생 타임라인 | loadDaewoon + Profection + Dasha + ZiweiDecadal (병렬) → align |
| `compatibility-v2` | 궁합 | loadSajuCompat + loadSynastry → mergeAxes |

### 12.4 Memory 전략

| 레벨 | 키 | 내용 | TTL |
|------|---|------|-----|
| Resource | `user:{id}` | 장기 성향, 선호 톤, 누적 패턴 | 장기, 주 1회 압축 |
| Thread | `user:{id}:thread:{id}` | 대화 맥락, 최근 질문 의도 | 세션, 30-50 메시지 |
| Snapshot | `reading:{id}:{date}` | Pack 결과 + evidence | 날짜 단위 캐시 |
