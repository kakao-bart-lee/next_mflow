# saju-core-lib Sync Policy

## 목적

이 문서는 `saju-core-lib`와 `next_mflow` 간 포팅, 동기화, 검증, 상태 보고 방식을 고정한다.

핵심 원칙은 다음 두 가지다.

1. 도메인 로직의 기준 구현은 `saju-core-lib`에 둔다.
2. `next_mflow` 포팅은 기준 구현을 추적 가능하게 옮기고 검증하는 과정으로 다룬다.

## 저장소 역할

### `saju-core-lib`

- 사주 계산 로직, 룰, 테이블 해석, 레거시 호환 규칙의 단일 소스 오브 트루스다.
- 도메인 동작 변경은 여기서 먼저 설계하고 수정한다.
- `next_mflow`에서 먼저 로직을 바꾸고 나중에 역이식하지 않는다.

### `next_mflow`

- `saju-core-lib`를 소비하거나 포팅하는 저장소다.
- 독자 구현이 필요하더라도 기준은 항상 특정 `saju-core-lib` 커밋 SHA에 연결되어야 한다.

## 포팅 순서

모든 도메인 로직 포팅은 아래 순서를 따른다.

1. `saju-core-lib`에서 기준 구현을 수정하거나 확정한다.
2. `saju-core-lib`에서 검증 게이트를 통과시킨다.
3. 그 시점의 커밋 SHA를 기준점으로 삼아 `next_mflow`에 포팅한다.
4. `next_mflow`에서 해당 포팅 단위의 테스트와 빌드를 통과시킨다.
5. `next_mflow` 커밋 메시지에 기준 `saju-core-lib` SHA를 남긴다.

## 커밋 추적 규칙

`next_mflow`의 포팅 커밋 메시지는 기준 SHA를 반드시 포함한다.

권장 형식:

```text
port(saju-core-lib@<sha>): <summary>
```

예시:

```text
port(saju-core-lib@8f7021b): migrate legacy compatibility detail bridge
```

이 규칙은 다음을 위해 필요하다.

- 어떤 포팅이 어떤 기준 구현을 반영했는지 추적
- 회귀 발생 시 비교 기준점 즉시 식별
- 향후 재포팅 또는 체리픽 시 출처 명확화

## 검증 게이트

### `saju-core-lib` 필수 게이트

포팅 단위 완료 전 아래 명령을 기본 게이트로 사용한다.

```bash
uv run pytest -q
npm run typecheck
npx vitest run
```

설명:

- `uv run pytest -q`: Python 기준 구현 전체 회귀 확인
- `npm run typecheck`: TypeScript 정적 타입 안정성 확인
- `npx vitest run`: TypeScript 런타임 회귀 확인

### `next_mflow` 필수 게이트

포팅 대상 모듈에 영향 있는 테스트와 빌드를 반드시 통과시킨다.

최소 원칙:

- 포팅 단위에 직접 대응하는 테스트 실행
- 앱 또는 패키지 빌드 성공 확인
- 회귀 가능성이 큰 경로는 관련 통합 테스트까지 포함

## 상태 보고와 근거 인용 규칙

상태 문서, 진행 노트, 비교 보고서에는 다음 규칙을 적용한다.

### 1. 현재 상태는 최신 실측 우선

- 현재 품질, 테스트 현황, 성숙도 평가는 최근 실행한 명령 결과를 우선 사용한다.
- 과거 계획 문서나 완료 보고서는 보조 근거로만 사용한다.
- 과거 문서의 수치가 현재와 다를 수 있으므로, 현재 상태 서술의 단독 근거로 쓰지 않는다.

예시:

- 현재 테스트 현황: `uv run pytest -q` 실측값 사용
- 리팩터링 완료 여부: `REFACTORING_PLAN.md`를 보조 근거로 사용

### 2. 과거 비교 아티팩트는 날짜를 붙여 인용

- 배치 비교 JSON, 이전 리포트, 스냅샷 결과는 생성 시점을 명시한다.
- 과거 비교 결과를 현재 품질의 확정치처럼 표현하지 않는다.

예시:

- `2026-01-20 배치 비교 기준 핵심 기둥 38/40`
- `2026-01-20 생성 아티팩트 기준 확장 항목 118/190`

### 3. TypeScript 검증은 정적/동적 범위를 분리해 표현

- `typecheck` 통과는 정적 타입 안정성 근거다.
- `vitest` 통과는 런타임 동작 근거다.
- 테스트 파일 수와 테스트 수가 적다면 검증 범위를 좁게 표현한다.

금지 표현:

- `TS 전체 정확도 검증 완료`
- `TypeScript가 Python보다 정확도까지 우세`

권장 표현:

- `TypeScript strict typecheck 통과`
- `선별된 레거시 호환 흐름 1개 파일, 4개 테스트 통과`
- `런타임 검증 폭은 아직 제한적`

## 변경 분리 규칙

문서/노트 변경과 코드 변경은 커밋을 분리한다.

분리 대상 예시:

- `.sisyphus`
- 운영 가이드
- 조사 메모
- 진행 문서
- 아키텍처 노트

코드 커밋에는 가능한 한 실행 로직, 테스트, 빌드 설정만 포함한다.

## 동시 수정 규칙

같은 모듈은 한 시점에 한 저장소에서만 편집한다.

운영 원칙:

- 기준 구현 수정 단계에서는 `saju-core-lib`만 편집
- 포팅 단계에서는 해당 모듈을 `next_mflow`에서 편집
- 동일 의미의 모듈을 양쪽에서 동시에 진화시키지 않는다

이 규칙은 다음 위험을 줄인다.

- 포팅 기준점 붕괴
- 양방향 드리프트
- 머지 충돌과 역추적 비용 증가

## 권장 운영 방식

포팅 작업을 시작할 때 아래 체크리스트를 따른다.

1. 변경 대상 모듈과 책임 저장소를 먼저 확정한다.
2. `saju-core-lib`에서 기준 구현과 테스트를 먼저 정리한다.
3. 기준 SHA를 기록한다.
4. `next_mflow`로 포팅한다.
5. 양쪽 저장소에서 검증 게이트를 통과시킨다.
6. 코드 커밋과 문서 커밋을 분리한다.
7. 상태 요약에는 최신 실측과 과거 아티팩트를 구분해서 적는다.

## 현재 기준 문구 템플릿

필요 시 아래 형식을 사용한다.

```text
구조와 운영성은 PHP 원본 대비 크게 개선되었다.
현재 Python 기준 구현은 최신 실측에서 전체 테스트를 통과했다.
TypeScript는 strict typecheck를 통과했고 선별된 레거시 호환 테스트는 통과했지만,
런타임 검증 범위는 아직 좁다.
외부 비교 수치는 2026-01-20 생성 배치 아티팩트 기준으로 해석한다.
```
