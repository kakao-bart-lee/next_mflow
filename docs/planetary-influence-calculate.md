어떤 방식으로 운세/운명을 결정하는지

핵심 파이프라인은 대체로 아래 순서입니다.
	1.	천체 위치 계산: 출생 순간의 태양·달·행성의 황경(0~360°) 등을 구함
	2.	**황도 12궁(트로피컬)**에 배치: 트로피컬은 계절 기준으로 **춘분점(0° 양자리)**를 기준점으로 둡니다.  ￼
	3.	하우스(12집) 배치: 하우스는 출생 시각/장소의 지평선·천정 등을 기준으로 황도를 나누는 여러 “하우스 시스템”(Placidus 등)으로 계산됩니다.  ￼
	4.	어스펙트(각도 관계): 행성 간 각도(예: 0°, 60°, 90°, 120°, 180° 등)를 계산해 “조화/긴장” 같은 해석을 붙입니다.  ￼
	5.	해석: “행성(무엇) + 별자리(어떻게) + 하우스(어디서) + 어스펙트(관계)”를 종합해 성향/테마/시기 등을 읽습니다.  ￼

결정 방식이 공개되어 있는지
	•	계산(천체 위치·하우스·각도): 공개된 천문/수학 기반 계산이며, 표준에 가까운 구현들이 많이 공유됩니다(예: Swiss Ephemeris).  ￼
	•	해석 룰(“이 배치는 이런 의미”): 전통서/현대서/학파별로 매우 다양하고, 일부는 저자/상담가의 노하우로 사실상 “비공식/준-독점”인 경우가 많습니다. 다만 기본 개념(차트 구성 요소)은 널리 공개되어 있습니다.  ￼


아래는 “최소 입력은 생년월일(YYYY-MM-DD), 그리고 사용자가 **시간/위치(출생지)**를 추가로 넣을 수 있는 UX”를 전제로, 앞서 정리했던 5개 체계(서양 트로피컬 / 헬레니즘 / 베다(조티시) / 사주 / 자미두수)를 제품·개발 설계 관점에서 최소 입력 / 필수 입력 / 출생시간 모를 때 대체 전략까지 구체화한 정리입니다.

공통 설계 프레임(5개 체계 모두에 적용)

1) 입력을 “등급”으로 취급하기

최소 입력(생년월일)만 받는 순간부터, 결과의 “정밀도”는 체계별로 크게 달라집니다. 그래서 **입력 누락이 아니라 ‘입력 등급’**으로 취급하는 게 구현/UX 모두 안정적입니다.
	•	L0 (최소): birth_date만 있음 (YYYY-MM-DD)
	•	L1: birth_date + birth_time 있음 (현지 시각인지/표준시인지 애매할 수 있음)
	•	L2: birth_date + birth_place 있음 (도시/위경도/국가 등)
	•	L3 (정밀): birth_date + birth_time + birth_place(+timezone) 있음

서양 점성 기준으로도 하우스/ASC 같은 핵심 값은 “시간+위치”에 의존합니다. (하우스는 “날짜”보다 “시간·장소” 의존이라는 요약이 매우 중요)  ￼

2) “계산값”과 “해석 리포트”를 분리
	•	계산 레이어: (천체력/달력/하우스/궁 배치/간지 변환 등) — 결정적(deterministic).
	•	해석 레이어: (의미 부여/길흉/성격 서술) — 룰 기반이긴 해도 학파·저자별 차이 큼.

UX는 입력 등급에 따라 계산 가능한 컴포넌트만 노출하고, 불가능한 컴포넌트는 “숨김/비활성화/범위 표시” 중 하나로 처리하는 게 좋습니다.

3) “출생시간 모름”은 3가지 모드로 나누면 실전에서 편합니다
	1.	시간 없음 → 하우스/ASC/궁 배치 완전 생략 모드
	2.	시간 없음 → ‘가정 시간(정오)’로 계산하되, 하우스는 생략 모드
	•	Astrodienst는 “출생시간 unknown으로 선택 시 **정오(noon)**로 잡고 하우스를 계산하지 않는다”는 방식을 FAQ로 명시합니다.  ￼
	3.	시간 없음 → 가능한 결과를 ‘범위/후보군’으로 계산 모드
	•	예: (서양/베다) 그날 가능한 ASC 후보와 시간 구간, (사주/자미) 12시진(2시간 단위) 후보 등

1) 서양 점성술(트로피컬 네이털)

최소 입력(L0: 생년월일만)로 가능한 것
	•	행성의 “별자리(사인)” 수준 일부: 태양/수성/금성/화성/목성…은 날짜 단위로 큰 틀이 나오지만,
	•	**달(Moon)**은 하루 안에도 별자리가 바뀔 수 있고
	•	ASC/MC/하우스는 시간/위치 없이는 원칙적으로 불가입니다.
	•	제품적으로는 L0에서 “하우스/ASC/MC/버텍스/포르투나(Part of Fortune) 등”은 숨김이 안전합니다. (하우스는 시간·장소 의존)  ￼

정밀 리포트에 “필수 입력”(L3)
	•	필수: 생년월일 + 출생시각(분 단위 권장) + 출생지(위도/경도) + 타임존(또는 타임존 자동추정)
	•	Astrodienst는 출생시간이 중요하며, ASC는 평균적으로 4분에 약 1도 움직일 수 있다고 설명합니다. 즉 “10~20분 오차”도 하우스/ASC에 크게 영향 가능.  ￼
	•	또한 입력 시간은 “출생지에서 손목시계에 찍히는 현지 시각”이며, 타임존/DST 보정은 시스템이 처리하는 방식이 일반적입니다.  ￼

L1/L2(시간만, 위치만)일 때의 설계 대응
	•	L1(시간만 있고 위치가 없음)
	•	행성 위치(지구중심 황경)는 계산할 수 있어도, ASC/하우스는 위치가 필요합니다(동쪽 지평선과 황도의 교점).  ￼
	•	UX: “행성 사인/각도(어스펙트) 중심 요약”만 제공 + “출생지 입력 시 ASC/하우스 추가됨” 안내.
	•	L2(위치만 있고 시간이 없음)
	•	추천 기능: “그날(0:00~24:00) ASC가 어떤 사인을 거치는지 시간대별로 보여주기(가능 후보군)”
	•	이는 유저에게 “시간만 알면 무엇이 달라지는지”를 직관적으로 보여줍니다.

출생시간 모를 때(대체 전략) — 추천 구현 순서
	1.	표준 전략(가장 안전): “시간 unknown”이면
	•	정오(12:00)로 임시 계산하되
	•	하우스/ASC는 계산/표시하지 않음
	•	Astrodienst가 동일한 원칙(unknown → noon, no houses)을 FAQ로 안내합니다.  ￼
	2.	범위 전략(부가 기능):
	•	달이 그날 어느 시각에 사인이 바뀌는지(가능한 달 사인 후보) 표시
	•	ASC는 “가능한 사인 후보와 시간 구간” 표시(위치가 있을 때만)
	3.	“솔라 차트(태양을 ASC처럼)” 옵션(선택 제공)
	•	일부 서비스는 출생시간 없이도 하우스를 만들기 위해 “태양 사인을 ASC로 두는 방식”을 씁니다(정통 네이털 하우스는 아니고 대체 표현).  ￼
	•	UX에서 “출생시간 미상 대체 방식” 라벨을 강하게 표시하는 게 좋습니다.

구현 라이브러리(계산 레이어)
	•	Swiss Ephemeris는 점성 계산에서 널리 쓰이는 고정밀 천체력 툴셋으로 소개됩니다.  ￼
	•	Python에서는 pyswisseph(바인딩) 사용이 흔합니다.  ￼
	•	“차트/하우스/어스펙트”까지 한 번에 다루려면 Kerykeion 같은 라이브러리를 검토할 수 있습니다.  ￼
	•	전통 점성 위주면 flatlib도 후보입니다.  ￼

2) 헬레니즘(전통 서양) 점성술

서양 네이털 차트를 공유하지만, “전통 기법(예: 프로펙션, 로트, 주야 구분)” 때문에 시간/ASC 의존도가 더 높게 설계되는 경우가 많습니다.

최소 입력(L0)로 가능한 것
	•	L0에서 할 수 있는 건 사실상 “서양 트로피컬의 L0 결과(행성 사인 일부)”와 유사합니다.
	•	다만 헬레니즘 핵심 기법들은 하우스(특히 Whole Sign Houses)와 ASC가 필요합니다. (헬레니즘 점성 자체가 ASC/하우스를 쓰는 “horoscopic astrology”로 설명됨)  ￼

정밀 리포트 필수 입력(L3)
	•	사실상 서양 네이털과 동일: 생년월일 + 출생시각 + 출생지(+타임존)

출생시간 모를 때(대체 전략)

헬레니즘에서 “시간 모름”은 특히 중요한데, 대표 기법인 Annual Profections 자체가 “ASC에서 시작해 1년마다 1사인씩 진행”하는 규칙으로 기록됩니다. 즉 ASC가 없으면 결과가 고정되지 않습니다.  ￼

그래서 앱 설계는 보통 아래 중 하나를 선택합니다.
	1.	전통 기법 비활성화(권장)
	•	시간 미상(L0/L2)에서는 “프로펙션/로트/주야(sext/sect) 기반 리포트”를 숨김 처리
	2.	후보군 전략(고급 옵션)
	•	“가능한 ASC 후보(시간대별) → 각 후보별 프로펙션 결과”를 다중 결과로 제공
	•	UX는 “12개 결과를 늘어놓기”보다, (위치가 있으면) “시간대 슬라이더”로 ASC 후보를 좁히게 하는 편이 낫습니다.
	3.	유저 입력 최소화형 프롬프트
	•	“낮/밤 출생(주야)”만이라도 추가 질문하면 전통 해석(sect)에서 큰 분기가 줄어드는 경우가 있어, 시간 모름일 때 1-bit 질문으로 자주 쓰입니다. (정밀 계산은 불가하지만 해석 분기엔 도움)

구현 참고(기법/정의 출처)
	•	Astrodienst Astrowiki의 Profection 정의(ASC 기준 1년 1사인 진행)가 설계 근거로 유용합니다.  ￼
	•	Hellenistic astrology가 ASC(“horoskopos”)와 그로부터 파생되는 12하우스를 사용한다는 개요는 Wikipedia 문서에 정리돼 있습니다.  ￼
	•	Astrodienst에는 Chris Brennan 글(헬레니즘 기법 소개)도 있어, “어떤 전통 기법을 제품에 넣을지” 범위 정의에 도움이 됩니다.  ￼

3) 베다 점성술(조티시, Jyotish)

최소 입력(L0)로 가능한 것
	•	L0(날짜만)로도 “태양 사인” 같은 아주 거친 값은 만들 수 있지만,
조티시 핵심인 라그나(ASC에 대응), 하우스, 나크샤트라, 다샤는 시간 의존이 큽니다.
	•	특히 Vimshottari Dasha는 “출생 시 달이 위치한 나크샤트라의 주(planet lord)에서 시작”하며, “달의 정확한 경도(나크샤트라 안에서 얼마나 진행했는지)”로 초기 기간 잔량까지 계산하는 방식이 정리돼 있습니다.  ￼
→ 따라서 시간 미상에서 다샤는 결과가 크게 흔들립니다.

정밀 리포트 필수 입력(L3)
	•	생년월일 + 출생시각 + 출생지(+타임존)
	•	추가로 조티시에서는 ayanāṃśa(아야남샤) 선택이 구현 옵션으로 들어갑니다(사이드리얼 보정 방식). Wikipedia는 여러 아야남샤 시스템이 있으며 Lahiri가 널리 쓰인다고 정리합니다.  ￼
→ 제품 설계로는 “기본값(Lahiri 등) + 고급설정에서 변경”이 일반적입니다.

출생시간 모를 때(대체 전략) — 추천 구현
	1.	다샤/라그나 기반 리포트 비활성화(안전)
	•	시간 없는 경우: “라그나/하우스/다샤” 탭을 잠그고 “시간이 필요함”을 명시
	2.	후보군 전략(시간 미상 but 위치가 있거나, 최소 타임존이라도 있는 경우)
	•	그날 24시간을 일정 간격(예: 30분/60분)으로 샘플링해서
	•	가능한 라그나 사인 구간
	•	달 나크샤트라 후보 및 다샤 시작 후보
를 “가능 후보”로 보여줍니다.
	3.	2시간 단위라도 받기(UX 최적)
	•	사용자에게 “정확한 분” 대신 “대략 시각대(새벽/오전/오후/밤)” 또는 “2시간 단위” 선택만 받아도 후보가 급격히 줄어듭니다.
	4.	시간 보정(레크티피케이션) 기능은 ‘고급/실험’으로 분리
	•	상용 서비스에 존재하지만, 구현 난이도/논쟁이 큽니다(수학적으로는 최적화 문제로 만들 수 있으나 룰 정의가 복잡).
	•	제품에서는 “선택 기능”으로 두고, 기본 플로우에 섞지 않는 편이 운영 안정적입니다.

구현 라이브러리 후보

	•	PyJHora(베다 점성 계산 기능을 폭넓게 구현한 파이썬 패키지).  ￼
	•	kunjara/jyotish(PHP).  ￼

4) 사주명리(사주팔자, Four Pillars / BaZi)

최소 입력(L0)로 가능한 것 — “3기둥 모드”가 자연스러운 이유

Four Pillars 문서 자체가 역사적으로 3기둥(년/월/일) → 이후 ‘출생시간’을 4번째 기둥으로 추가됐다고 설명합니다. 즉 시간 미상이라면 “3기둥 리포트”가 자연스러운 폴백입니다.  ￼

그래서 L0에서 앱이 제공할 수 있는 안전한 출력은:
	•	년주/월주/일주(3기둥) 계산 + 오행 분포(시간 기둥 제외한 범위)

정밀 리포트 필수 입력(L3)
	•	생년월일 + 출생시각 + (권장) 출생지/타임존
	•	Hong Kong Observatory는 간지(천간/지지) 체계에서 “시간(시辰, shichen)이 지지로 이름 붙었고, 년/월/일/시의 간지가 모여 8글자(Eight Characters)”를 이룬다고 설명합니다.  ￼
	•	Earthly Branches 표는 각 지지가 담당하는 “2시간 구간”을 명시합니다(예: 子 23:00–01:00 …).  ￼
→ 즉 사주는 출생시간을 “분 단위”로 몰라도, 최소한 2시간 단위 시진만 알아도 4기둥이 확정됩니다(정확한 분까지는 확장 옵션).

출생시간 모를 때(대체 전략) — 추천 구현
	1.	3기둥 리포트 제공(기본 폴백)
	•	“시간 미상 → 시주 미포함”을 명시하고, 가능한 해석 범위를 제한
	2.	12개 시진 후보(子~亥) 동시 계산(고급 폴백)
	•	12개 시주 후보를 전부 계산해
	•	“변하지 않는 부분(년/월/일)”
	•	“시주에 따라 달라지는 부분”
을 비교 UI로 보여줍니다.
	3.	최소 추가 질문 1개로 후보 대폭 축소
	•	“새벽/오전/오후/밤” 같은 4분할 질문만 해도 시진 후보가 크게 줄어듭니다.
	4.	(선택) 시주 추정(이벤트 기반)
	•	사용자에게 큰 사건 연도(졸업/취업/결혼/이사 등)를 받아 “대운/세운 매칭”을 시도할 수 있지만, 룰 정의/학파 차이가 크므로 실험 기능으로 격리 권장.

	•	(보조) 간지/지지·시진 정의 참고: Hong Kong Observatory, Earthly Branches(Wikipedia).  ￼

5) 자미두수(Zi Wei Dou Shu)

최소 입력(L0)로 가능한 것
	•	Wikipedia는 자미두수가 “순수한 음력 달력과 밤하늘 위치에 기반”한다고 요약합니다.  ￼
	•	실무적으로는 출생시간 의존도가 매우 높아서, L0(날짜만)로 “자미 명반(12궁 성요 배치)”을 안정적으로 만들기 어렵습니다.

즉 제품 설계 관점에서:
	•	L0에서 자미두수 탭은 “개요/입력 유도” 수준으로 두고,
	•	“시간 입력(최소 2시간 단위라도)”이 들어오는 순간부터 본 리포트를 여는 방식이 흔합니다.

정밀 리포트 필수 입력(L3에 준함)
	•	생년월일(양력 또는 음력) + 출생시간 + 성별이 보통 필요합니다.
	•	대표 오픈소스 라이브러리 iztro의 README도 입력으로 (1) 생일(양력/음력), (2) 출생시간, (3) 성별을 명시하고, 출력으로 12궁 성반 데이터 및 운한(대/소/유년/월/일/시) 등을 제공한다고 설명합니다.  ￼
	•	또한 iztro는 “유파가 많아(사화/밝기 등) 차이가 있으니 설정/플러그인으로 확장”하는 설계를 언급합니다.  ￼

출생시간 모를 때(대체 전략)
	1.	최소 단위로 ‘시진(2시간 구간)’ 선택 UI 제공(권장)
	•	자미는 분 단위보다 “시진” 입력으로도 많이 사용되므로, UX에서 HH:MM보다 “子~亥(12구간)”을 먼저 제시하면 이탈률이 줄어듭니다.
	2.	12개 시진 전부 계산해 후보군 제공(고급)
	•	사용자가 시간을 전혀 모르면 12개 명반 후보를 모두 생성하고,
	•	핵심 차이(명궁/신궁/주성 배치 등)만 요약 비교
	•	“어느 쪽이 더 본인과 맞는지” 선택하게 하는 흐름
	3.	(선택) 시간 추정(레크티피케이션) 기능
	•	사주와 마찬가지로 구현 난이도/논쟁성이 있어, 메인 플로우와 분리 권장.

구현 라이브러리 후보
	•	iztro(JavaScript/TypeScript).  ￼
	•	iztro-py(Python).  ￼
